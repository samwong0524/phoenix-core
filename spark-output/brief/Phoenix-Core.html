Phoenix-Core

产品改版 · 2026-06-30

DESIGN BRIEF

业务背景 Business Context

当前业务处于什么阶段，触发本次设计的原因

Phoenix-Core 多 Agent 蜂群系统前端已完成 4 阶段商用化改版（TSC 0、570 tests、81% 覆盖），但启发评估发现 48 个体验问题（7 blocker / 41 major），首次使用引导和响应式适配为系统性缺失，驱动本次改版。

业务目标 Business Goal

要推动哪些可量化的业务结果

  • 新用户 5 分钟内完成首次 Agent 交互
  • Blocker 级 UX 问题清零（7 项）
  • 错误反馈模式统一为 1 种标准组件
  • 核心页面移动端可用率 > 80%

用户 User

主要受众是谁，核心使用场景

  • AI 研发工程师——日常创建/监控多 Agent 协作任务
  • 工作流开发者——构建和调试 Workflow/Pipeline DAG
  • 运维人员——实时观测 Agent 运行状态和系统指标
  • 首次接触多 Agent 聊天范式的新用户

设计策略 Design Strategy

从哪些维度切入，用什么方法应对目标

信息架构 IA

核心操作提至首屏，减少结算路径层级。

  • 结算入口上移到首屏
  • 精简冗余跳转层级
  • 分类重组贴近心智

对应 B2 跳出率↓20%；用户低容忍繁琐

交互设计

结算压缩到 3 步，错误校验前置。

  • 地址 / 支付合并一屏
  • 字段实时校验提示
  • 进度条常驻顶部

对应 B1 转化 45%→58%

情感化设计

支付完成正向动效，加购即时反馈。

  • 支付成功动效 + 音效
  • 加购浮层确认态
  • 异常态温和提示

对应 B4 客诉↓30%

设计标准 Design Criteria

体验维度的判断依据

  • 定量移动端可用率 > 80%
  • 定量核心任务操作步骤 ≤ 3
  • 定量触摸目标 ≥ 44px
  • 定性断点切换无布局崩坏

边界约束 Constraints

时间、技术、资源等硬性限制

  • Next.js 16 + React 19 + Tailwind v4 不变
  • Zustand + Immer 状态管理架构保留
  • 后端 SSE / REST API 接口不变
  • framer-motion（Corporate 调性）延续
  • 8 周完成改版

不做什么 Out of Scope

本次明确排除的内容，防止范围蔓延

  • 后端 Agent 运行时架构不改
  • 数据库 schema 不变
  • OAuth 认证流程不改
  • 不引入新的状态管理库
  • i18n 方案（React Context + Cookie）不变
