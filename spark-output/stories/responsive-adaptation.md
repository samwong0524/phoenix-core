## Story 1

**陈昊在手机上也能顺畅查看 Agent 聊天进度**

- **Persona**：陈昊，28 岁 AI 研发工程师，日常在工位和会议室之间切换，经常需要在手机上查看 Agent 任务执行进度
- **JTBD**：当陈昊在开会或通勤时，陈昊想快速打开手机查看 Agent 群组的执行状态和最新回复，从而决定是否要介入调整。
- **关联策略维度**：多端适配
- **优先级**：P0
- **关键假设标记**：⭐（测试假设：移动端可用率 > 80% 的 Brief 目标能否达成）

**Story 主体**：

作为陈昊，
陈昊想在手机浏览器里打开 Phoenix-Core 的 IM 页面，看到 Agent 群组最新的对话内容和执行状态，
从而不用跑回工位就能判断任务是否正常推进。

**这个 Story 在用户那一侧长什么样**：

陈昊从会议室出来，掏出手机打开 IM 页面。左侧栏自动收起为汉堡菜单，中间聊天区占满屏幕，最新的 Agent 消息清晰可读。陈昊手指一滑就看到 3 条新回复，点一下右上角按钮还能展开任务监控面板的摘要。整个过程不到 10 秒，陈昊心里有数了。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊在 375px 宽的手机上能看到完整的聊天记录，不需要横向滚动
- 陈昊点击汉堡菜单能在 0.3 秒内展开侧栏（Agent 列表 + 目录）
- 陈昊能触摸滚动聊天消息，触摸目标不小于 44px
- 陈昊能在手机上看到最新的 Agent 活动状态文字（"深度思考中"等已翻译为当前语言）

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：IM 主页面 (im/page) / 汉堡菜单 / 聊天消息列表
- 涉及组件：汉堡菜单按钮 / 滑出抽屉 / 聊天消息气泡 / 活动状态条
- 涉及状态：初始（加载中）/ 正常（消息列表）/ 空（无消息）
- 涉及交互模式：responsive-layout / drawer-navigation / touch-scroll

**测试的假设**：

移动端可用率 > 80% 的核心场景——IM 页面是陈昊使用频率最高的页面，如果手机端 IM 不可用，整个改版目标无法达成。

**我的诚实看法**：

IM 页面是 1060 行的复杂组件，内含 SSE 流式更新 + ResizeObserver + 拖拽事件。响应式改造不只是加 media query，需要重构布局策略（从固定三栏到条件渲染），工程量是 6 个 Story 里最大的。但它是核心，不能跳过。

## Story 2

**陈昊的三栏布局在平板上自动折叠为侧栏抽屉**

- **Persona**：陈昊，28 岁 AI 研发工程师，有时用 iPad 在沙发上查看和调整 Agent 配置
- **JTBD**：当陈昊用平板（768–1024px）操作时，陈昊想在不缩放的情况下看清聊天内容，同时能随时调出侧栏切换 Agent 群组。
- **关联策略维度**：多端适配
- **优先级**：P0

**Story 主体**：

作为陈昊，
陈昊想用 iPad 打开 IM 页面时，侧栏自动隐藏为可滑出的抽屉，聊天区占据大部分宽度，
从而阅读体验接近桌面端而不需要双指缩放。

**这个 Story 在用户那一侧长什么样**：

陈昊坐在沙发上打开 iPad，IM 页面的左侧栏已自动折叠。中间聊天区宽度刚好，消息排版舒服。陈昊想换个 Agent 群组，从左边缘向右一滑，侧栏像抽屉一样拉出来，陈昊点了另一个群组，抽屉自动收回，新群组的消息出现在聊天区。陈昊觉得"跟在电脑上差不多"。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊在 768px 宽的平板上看到侧栏默认隐藏，聊天区占满可用宽度
- 陈昊从左边缘向右滑动能展开侧栏（touch gesture 或按钮触发）
- 陈昊点击侧栏中的 Agent 群组后，侧栏自动收回
- 陈昊的触摸操作（滑动、点击）全部在 44px 触摸目标内

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：IM 主页面 / 侧栏抽屉 / Agent 群组列表
- 涉及组件：抽屉组件 / 触摸手势识别 / 群组列表项 / 遮罩层
- 涉及状态：抽屉关闭 / 抽屉打开 / 群组切换中
- 涉及交互模式：drawer / swipe-gesture / responsive-breakpoint

**测试的假设**：

n/a

**我的诚实看法**：

平板是桌面到手机的过渡地带。如果只做好桌面 + 手机两端，平板用户的体验会"两边都不靠"。这个 Story 的关键决策是：断点设在哪里（768px vs 1024px），以及抽屉是覆盖式（overlay）还是推移式（push）。建议先做 mobile-first 的 overlay 抽屉，再向上兼容。

## Story 3

**陈昊在手机上编辑 Workflow 时不再丢失节点**

- **Persona**：陈昊，28 岁 AI 研发工程师，偶尔需要在手机上快速查看和微调 Workflow 配置
- **JTBD**：当陈昊在手机上打开 Workflow 编辑器时，陈昊想至少能查看节点配置和连线关系，从而确认工作流逻辑正确而不需要赶回工位。
- **关联策略维度**：多端适配
- **优先级**：P0

**Story 主体**：

作为陈昊，
陈昊想在手机上打开 Workflow 页面时看到画布的缩略视图（只读），需要编辑时切换到简化的表单模式，
从而在手机上也能确认和微调 Workflow 配置。

**这个 Story 在用户那一侧长什么样**：

陈昊收到同事消息说"Workflow 里有个节点参数不对"。陈昊掏出手机打开 Workflow 页面，看到画布自动缩小为只读概览，节点和连线一目了然。陈昊点了一下有问题的节点，底部弹出一个 sheet 面板显示该节点的配置表单。陈昊改了一个参数，点"保存"，系统提示"已保存"。陈昊松了口气，不用跑回工位了。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊在手机上看到 Workflow 画布自动缩放为只读概览（fit-to-screen）
- 陈昊点击节点后，底部弹出配置 sheet（不超过屏幕 60% 高度）
- 陈昊能在 sheet 表单中修改参数并保存，保存后有明确成功反馈
- 陈昊的双指缩放手势能放大/缩小画布（触摸事件处理）

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：Workflow 编辑器 (workflow/page) / WorkflowCanvas / 节点配置面板
- 涉及组件：画布缩放控件 / 底部 sheet / 表单 / 节点缩略图
- 涉及状态：只读概览 / 节点选中 / 编辑模式 / 保存中 / 保存成功
- 涉及交互模式：pinch-zoom / bottom-sheet / responsive-canvas

**测试的假设**：

n/a

**我的诚实看法**：

Workflow Canvas 是三个固定面板（NodePalette 200px + canvas flex + Properties 280px），最小宽度 780px。手机上完整编辑不现实。"只读概览 + 简化编辑 sheet"是务实方案。但要注意：如果陈昊只是看，不需要改，那 Story 3 的编辑功能就是过度设计。建议先做只读（P0），编辑能力放 P1。

## Story 4

**陈昊误删 Agent 节点后能一键撤销**

- **Persona**：陈昊，28 岁 AI 研发工程师，经常拖拽和调整 Workflow 节点布局，偶尔手滑按到 Delete 键
- **JTBD**：当陈昊在编辑 Workflow 时不小心删除了一个精心配置的节点，陈昊想立刻恢复它，从而不丢失已经填好的复杂参数。
- **关联策略维度**：撤销/恢复与破坏性操作防护（audit 机会点 3）
- **优先级**：P0

**Story 主体**：

作为陈昊，
陈昊想在误删 Workflow 节点后立刻看到"撤销"提示，点一下就能恢复，
从而不重新手动配置一个包含十几个参数的 Agent 节点。

**这个 Story 在用户那一侧长什么样**：

陈昊在拖拽节点时手指一滑按到了 Delete 键，节点消失了。陈昊心里一紧——那个节点配了 15 分钟。但右下角弹出一个 toast："节点已删除"，旁边有个"撤销"按钮。陈昊赶紧点"撤销"，节点带着所有配置原样回来了。陈昊长出一口气。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊删除节点后 3 秒内看到 toast "节点已删除 · 撤销"
- 陈昊点击"撤销"后节点原样恢复（含所有配置数据）
- 陈昊按 Ctrl+Z 也能触发撤销（键盘快捷键）
- 陈昊连续撤销最多回退 20 步操作

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：Workflow 编辑器 (workflow/page) / WorkflowCanvas
- 涉及组件：Toast（带操作按钮）/ Undo 栈（Zustand temporal middleware）
- 涉及状态：节点正常 / 节点已删除 / 撤销中 / 撤销成功
- 涉及交互模式：undo-redo / toast-with-action / keyboard-shortcut

**测试的假设**：

n/a

**我的诚实看法**：

这是 7 个 Blocker 里修复成本最低的一个（audit-4 标记为 quick-win）。Zustand + Immer 天然支持 temporal middleware，加 undo/redo 栈大概 50 行代码。但这个 Story 的真正价值不只是 Workflow——Skills 删除（audit-7）、Stop All（audit-11）、Pipeline Review（audit-28）都需要同样的"操作确认 + 撤销"模式。建议做成全局通用组件。

## Story 5

**陈昊第一次打开系统就知道各面板是干嘛的**

- **Persona**：陈昊，28 岁 AI 研发工程师，第一次使用 Phoenix-Core 多 Agent 蜂群系统
- **JTBD**：当陈昊第一次登录 Phoenix-Core 时，陈昊想快速理解界面上每个区域的作用，从而在 5 分钟内发出第一条消息并看到 Agent 响应。
- **关联策略维度**：首次使用引导体系（audit 机会点 1）
- **优先级**：P1

**Story 主体**：

作为陈昊，
陈昊想第一次打开 IM 页面时看到简短的面板说明，告诉陈昊左边是 Agent 群组、中间是聊天、右边是任务监控，
从而不用自己摸索每个区域的功能。

**这个 Story 在用户那一侧长什么样**：

陈昊 OAuth 登录成功后进入 IM 页面。页面上方出现一个半透明引导层，高亮左侧栏并写着"这里是 Agent 群组——选择或创建你的协作团队"。陈昊点"下一步"，中间聊天区被高亮："在这里跟 Agent 对话，输入 @ 可以附加技能"。陈昊再点一下，右侧任务监控区亮起："实时查看 Agent 的工作进度和产出"。最后引导层消失，陈昊看到输入框里的提示"试试输入 @ 附加技能，Ctrl+Enter 发送"，陈昊知道怎么开始了。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊首次登录后看到 3–4 步引导覆盖三个主面板
- 陈昊可以点"跳过"直接关闭引导
- 陈昊引导结束后的 30 秒内能在输入框看到 @skill 提示
- 陈昊可以在设置里选择"重新播放引导"

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：IM 主页面 (im/page) / IMShell / 登录页跳转
- 涉及组件：引导层（react-joyride 或自建）/ 高亮蒙层 / 步骤气泡 / 跳过按钮
- 涉及状态：首次访问 / 引导进行中 / 引导完成 / 引导跳过
- 涉及交互模式：onboarding-tour / coachmark / progressive-disclosure

**测试的假设**：

n/a

**我的诚实看法**：

Audit-1 标记这是 Blocker（无首次使用引导），但 Brief 的策略维度只选了"多端适配"，引导体系归在"机会点 1"中未被选为本次策略维度。Story 优先级降为 P1 合理，但建议在改版工程中作为"顺手做"的 quick-win——引导层本身开发量不大（react-joyride 接入 + 4 步文案），但对新用户体验影响巨大。

## Story 6

**陈昊看到一条统一的错误提示而不是 5 种不同反馈**

- **Persona**：陈昊，28 岁 AI 研发工程师，在使用多个页面时会遇到各种操作失败的情况
- **JTBD**：当陈昊的操作失败时（保存、删除、连接断开），陈昊想立刻知道发生了什么以及能不能重试，从而不会困惑于"刚才到底成功了没有"。
- **关联策略维度**：错误反馈标准化（audit 机会点 4）
- **优先级**：P1

**Story 主体**：

作为陈昊，
陈昊想在任何页面遇到操作失败时看到同一种风格的错误提示（右上角 toast），附带"重试"或"关闭"按钮，
从而不用猜测"这次没有反应是成功了还是失败了"。

**这个 Story 在用户那一侧长什么样**：

陈昊在 Workflow 页面点"运行"，后台 API 返回 500 错误。右上角滑出一条红色 toast："运行失败——服务端异常，请稍后重试"，旁边有个"重试"按钮。陈昊点"重试"，toast 消失，系统重新发起请求。陈昊切到 Models 页面改 API Key 保存失败，同样右上角滑出红色 toast 提示。陈昊发现所有页面的错误提示长一样，心里踏实了——至少知道"没弹 toast 就是没出错"。

**这个 Story 算完成的标准**（可观察、可验证）：

- 陈昊在 IM / Workflow / Pipeline / Skills / Models / Settings 6 个页面看到同样风格的 toast 错误提示
- 每条 toast 有"关闭"按钮（X）和自动消失（5 秒）
- 可重试的错误（网络/API）显示"重试"按钮
- 不再出现 alert() / console.error 静默 / inline text 三种混杂模式

**设计触点**（让 Flow Web/Mobile 直接消费）：

- 涉及屏：全站所有页面
- 涉及组件：统一 Toast 组件 / 错误边界 / 重试按钮
- 涉及状态：操作成功 / 操作失败（可重试）/ 操作失败（不可重试）/ 连接断开
- 涉及交互模式：toast-notification / error-boundary / retry-pattern

**测试的假设**：

n/a

**我的诚实看法**：

当前系统有 5 种错误反馈模式（原生 toast 无关闭、console.error 静默、alert() 阻塞、inline text、空 catch），分布在至少 6 个页面。统一为一种需要：① 创建全局 Toast 组件 ② 逐页替换 catch 块 ③ 替换 alert()。工程量中等但影响全站——建议作为"基础设施"任务，在所有页面级改版之前完成。

**参考产品**：

Notion 的 toast 系统做得好——右上角统一出现，3 秒自动消失，成功绿/失败红/警告黄三色区分，网络断开时顶部出现持久 banner 而非 toast。Phoenix-Core 可以参考这个模式。
